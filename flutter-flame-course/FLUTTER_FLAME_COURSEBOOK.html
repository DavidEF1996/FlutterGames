C00 · Block 0 · Preparation

Environment, assessment, and workflow

Know which tool owns each responsibility, prove that the toolchain works, and adopt a repeatable development loop.

Learning objectives

  • Explain the roles of Dart, Flutter, Flame, the editor, and a target device.
  • Diagnose the installed toolchain without treating every warning as a blocker.
  • Use a small edit-format-analyze-run-test workflow.

Mental model: a stack, not one giant program

Dart is the language. Flutter is the SDK, rendering framework, and cross-platform toolchain. Flame is a package running inside Flutter that supplies a game loop, a component tree, input helpers, sprites, collisions, cameras, and related modules. VS Code or Android Studio edits the files; it does not replace the SDK. Finally, a browser, desktop process, emulator, or phone executes the compiled application.

Worked example: deciding whether a warning matters

Suppose flutter doctor reports that Chrome and Windows are ready, while Android licenses are missing. A Windows-first exercise is not blocked. Record Android as future setup work and continue. A useful diagnostic separates current target blockers from features you are not using yet.

Common pitfall: fixing the entire universe first

Beginners often spend hours making every doctor row green, install several IDEs, and update unrelated packages before running one frame. Choose one initial target. Stabilize only that path, record exact versions, and avoid upgrades in the middle of a chapter unless an error requires them.

Checkpoint

  1. Can a normal Flutter application exist without Flame? Can a Flame game exist without Flutter?
  2. Why can a doctor warning be real but irrelevant?
  3. What is the difference between static analysis and running the application?
Answer and discussion

Flutter does not require Flame; Flame is built on Flutter and therefore does require it. A warning matters only if it affects a platform or tool you currently intend to use. Static analysis inspects source code for type, lint, and API problems without executing the game, while running exercises the actual build and runtime path.

Practice: create an environment dossier

Run the four diagnostic commands. Record the Flutter and Dart versions, chosen first target, visible devices, blockers, and deferred warnings. Then write a five-step workflow you will follow after each meaningful change. Do not create the game project until you can explain the stack above.

Takeaways

  • Toolchain layers have separate responsibilities.
  • One working target is enough to begin.
  • Versions and diagnostic decisions belong in the project log.

C01 · Block 0 · Preparation

Essential Dart and Flutter foundations

Use types to protect game rules, objects to name responsibilities, async code for loading, and widgets as projections of state.

Learning objectives

  • Read null-safe Dart classes, enums, collections, functions, and mixins.
  • Understand why asset loading returns a Future.
  • Distinguish persistent state from the widget tree that displays it.

Mental model: state, behavior, projection

A game stores state—health, score, position, phase—and exposes behavior that changes that state legally. Flutter widgets project some of that state onto a screen. Rebuilding a widget is normal; destroying the only copy of the game session during a rebuild is not. Keep rules in plain Dart when possible, then let Flutter or Flame present them.

Types

Make impossible values harder to represent. An enum is clearer than strings such as "ded" and "dead".

Objects

Bundle related state and behavior. Prefer meaningful methods over unrestricted public mutation.

Async

A Future represents a value that will arrive later, such as a decoded image or save file.

lib/domain/player_stats.dartDart
enum PlayerCondition { ready, hurt, defeated }

class PlayerStats {
  PlayerStats({this.maxHealth = 3}) : health = maxHealth;

  final int maxHealth;
  int health;
  int score = 0;

  PlayerCondition get condition {
    if (health == 0) return PlayerCondition.defeated;
    if (health < maxHealth) return PlayerCondition.hurt;
    return PlayerCondition.ready;
  }

  void collect({int points = 10}) {
    if (condition == PlayerCondition.defeated) return;
    score += points;
  }

  void takeDamage(int amount) {
    health = (health - amount).clamp(0, maxHealth).toInt();
  }
}

Future<PlayerStats> loadPlayer() async {
  await Future<void>.delayed(const Duration(milliseconds: 100));
  return PlayerStats();
}

Worked example: why a method beats a public assignment

If any component can run health = -900, every reader must defend against nonsense. takeDamage() centralizes clamping and later becomes the natural place for invulnerability, sound, or events. Encapsulation is not ceremony; it keeps rules consistent.

Common pitfall: silencing null safety

The postfix ! means “I promise this is not null.” It does not make a value safe; it moves failure to runtime. Prefer required constructor parameters, early returns, nullable access, or initialization in a lifecycle method that guarantees the value exists.

Checkpoint

  1. Why is PlayerCondition safer than a free-form string?
  2. What does await do to the current async function?
  3. Why should game rules not depend directly on a particular text widget?
Answer and discussion

An enum restricts values and enables exhaustive tooling. await suspends that async function until the future completes without blocking the entire application thread. Rules outlive individual widgets and should be testable without rendering a screen.

Practice: model a collectible session

Create a pure Dart GameSession with score, remaining time, and a phase enum. Add methods for collecting an item, ticking time, winning, losing, and resetting. Write five assertions covering boundaries. Bonus: expose read-only getters while keeping mutation private.

Takeaways

  • Types and enums communicate rules before the game runs.
  • Async loading belongs in explicit lifecycle boundaries.
  • Flutter widgets display state; they do not need to own all of it.

GameWidget, FlameGame, and World

Mental model: bridge, director, stage

GameWidget is the Flutter bridge: it gives Flame screen space, focus, overlays, and lifecycle integration. FlameGame owns the loop and root component tree. Worked example: where should a pause menu live?

The moving player belongs in World. A pause menu made from normal Flutter controls belongs above GameWidget as an overlay. The score could be a Flame HUD component or a Flutter widget depending on the desired layout and accessibility. Ownership follows coordinate space and responsibility.

Common pitfall: constructing the game inside every build

A Flutter build() method may run many times. Creating CourseGame() there without preserving the instance can reset the world unexpectedly. Create the instance once in state, provide it from outside the frequently rebuilt subtree, or use GameWidget.controlled deliberately.

Checkpoint

  1. Which object owns the game loop?
  2. Where do enemies normally live?
  3. Why is GameWidget valuable even when the game fills the screen?
Answer and discussion

FlameGame owns the loop. Enemies normally live under World. GameWidget integrates the game with Flutter, making overlays, navigation, lifecycle, layout, and focus available even for a full-screen game.

Practice: shape playground

Create a project, pin the resolved versions, and show three primitive components with different positions, colors, sizes, and anchors. Resize the window. Record what remains stable and what changes. Keep the game instance alive across a parent widget rebuild.

Takeaways

  • GameWidget embeds; FlameGame coordinates; World contains.
  • A camera renders a world rather than the world painting itself directly.
  • Start with shapes so asset problems cannot hide structural problems.

C03 · Block 1 · Flame essentials

Game loop, dt, and lifecycle

Advance the simulation by elapsed time, load once at the right boundary, and clean up what a component owns.

Learning objectives

  • Explain the update-render cycle and delta time in seconds.
  • Choose appropriate lifecycle callbacks for loading, mounting, resizing, and removal.
  • Implement frame-rate-independent movement.

Mental model: simulate, then present

During an update, the game advances state by the elapsed time dt. During render, it draws the current state. At 60 updates per second, dt is roughly 0.0167; after a brief stall it may be larger. Movement must therefore be speed per second multiplied by elapsed seconds, not “pixels per frame.”

input arrives ↓ update(dt): state changes → positions, timers, AI, collisions ↓ render(canvas): current state becomes pixels ↓ display frame → repeat
CallbackUse it forAvoid
onLoadOne-time async setup, assets, initial children.Per-frame calculations.
onMountWork requiring a mounted parent/tree.Assuming it runs only once forever.
update(dt)Movement, timers, rules that advance.I/O, decoding images, allocating constantly.
onGameResizeReacting to canvas-size changes.Resetting the whole session.
onRemoveUnsubscribing and releasing owned resources.Leaving external listeners active.
lib/game/moving_block.dartDart
import 'package:flame/components.dart';

class MovingBlock extends RectangleComponent {
  MovingBlock({
    required super.position,
    required this.velocity,
  }) : super(size: Vector2.all(36));

  final Vector2 velocity; // world units per second

  @override
  void update(double dt) {
    super.update(dt);
    position += velocity * dt;
  }
}

Worked calculation

A ship moving at 120 units per second receives dt = 0.016. It advances 120 × 0.016 = 1.92 units. If the next frame takes twice as long, it advances about twice as far. Over real time, speed remains stable.

Common pitfall: hiding expensive work in update

Loading an image, parsing JSON, or creating large lists inside update() repeats that cost every frame. Load at a lifecycle boundary, cache what is reusable, and let update mutate small existing values. Also guard extreme dt where your rules require it; returning from the background can produce a large gap.

Checkpoint

  1. Why is position.x += 2 frame-rate dependent?
  2. Can onMount run again after removal and re-addition?
  3. Should render decide whether the player has won?
Answer and discussion

Two units per frame becomes faster as frame count increases. A component may mount again, so mounting is not synonymous with one-time construction. Winning is a state update; render should display the result rather than create it.

Practice: compare two movers

Create one component that moves by a constant amount per frame and another that uses velocity multiplied by dt. Temporarily simulate different timing or resize/stress the app. Explain the observed difference, then remove the incorrect mover. Add pause/resume controls and verify no giant jump occurs.

Takeaways

  • dt is elapsed seconds, not a target frame count.
  • Lifecycle methods establish safe ownership boundaries.
  • Update changes state; render shows state.

C04 · Block 1 · Flame essentials

Flame Component System and structure

Compose behavior into a tree, assign one clear owner to each object, and communicate through small meaningful interfaces.

Learning objectives

  • Read the Flame component tree and lifecycle propagation.
  • Choose parent-child relationships based on ownership and transforms.
  • Avoid global lookups and circular dependencies.

Mental model: a scene tree is an ownership tree

A component can own children. Children inherit the parent's transform and lifecycle context. A health bar attached to an enemy can be its child because it moves and disappears with that enemy. A global score HUD should not be a child of the player because a player respawn must not destroy the session UI.

FlameGame ├── World │ ├── Player │ │ ├── Sprite │ │ └── Weapon │ ├── EnemySpawner │ └── Level └── Camera └── Viewport ├── ScoreText └── PauseButton
lib/game/player.dartDart
import 'dart:ui';

import 'package:flame/components.dart';
import 'package:flame/game.dart';

class CourseGame extends FlameGame {
  int score = 0;

  void collectCoin() {
    score += 10;
  }
}

class Player extends PositionComponent
    with HasGameReference<CourseGame> {
  Player({required super.position})
      : super(size: Vector2.all(40), anchor: Anchor.center);

  @override
  Future<void> onLoad() async {
    await super.onLoad();
    await add(
      CircleComponent(
        radius: 6,
        position: size / 2,
        anchor: Anchor.center,
        paint: Paint()..color = const Color(0xffffb02e),
      ),
    );
  }

  void collect() {
    game.collectCoin();
  }
}

Worked decision: child or sibling?

A sword that rotates with the player is naturally a child. A projectile that continues after the player is removed should be added to the world as a sibling. Ask two questions: “Should it inherit this transform?” and “Should it share this lifetime?” If either answer is no, reconsider the parent.

Common pitfall: a god component

A single game class that moves every enemy, draws every object, plays every sound, and saves every setting becomes difficult to test or change. Split by responsibility, but do not create empty abstractions for their own sake. A good component owns a coherent piece of state and behavior.

Checkpoint

  1. Why might a weapon be a child of a player?
  2. What does priority affect?
  3. Why are meaningful game methods preferable to editing unrelated fields?
Answer and discussion

A child can inherit position and rotation and share removal. Priority controls ordering among siblings, including update/render order where relevant; it is not a substitute for a sensible tree. Methods such as collectCoin() protect rules and reduce coupling.

Practice: refactor a small scene

Build a player with a child marker, three independent collectibles, and a HUD attached to the camera viewport. Remove and re-add the player. Confirm which objects disappear, which remain, and why. Draw the final component tree before writing a second version.

Takeaways

  • Parenthood expresses transform and lifetime ownership.
  • Small components are useful when they own real responsibility.
  • Communicate through game concepts, not arbitrary shared mutation.

C05 · Block 1 · Flame essentials

2D mathematics, coordinates, and transformations

Turn positions into directions, directions into motion, and local shapes into a coherent world.

Learning objectives

  • Use Vector2 for positions, offsets, directions, and velocities.
  • Distinguish local, parent, world, camera, and screen coordinates.
  • Predict how anchor, scale, and angle affect a component.

Mental model: a vector has context

Vector2(3, 4) may be a position, an offset, a velocity, or a size. The numbers alone do not say which. Names and coordinate space give the vector meaning. Subtracting two positions produces an offset. Normalizing that offset produces a unit direction. Multiplying by speed produces velocity; multiplying velocity by dt produces this frame's displacement.

target position - current position = offset offset / offset length = unit direction unit direction × units per second = velocity velocity × delta seconds = frame displacement
PropertyMeaningFrequent surprise
positionAnchor location in the parent's coordinates.It is not always the top-left corner.
anchorReference point inside the component.Changing it appears to move an unchanged position.
angleRotation in radians around the anchor.Positive rotation follows the canvas coordinate convention.
scaleMultiplier applied around the anchor.Children inherit parent transforms.
lib/game/seeker.dartDart
import 'dart:math' as math;

import 'package:flame/components.dart';

class Seeker extends PositionComponent {
  Seeker({
    required super.position,
    required this.target,
    this.speed = 140,
  }) : super(size: Vector2.all(24), anchor: Anchor.center);

  Vector2 target;
  final double speed;

  @override
  void update(double dt) {
    super.update(dt);

    final offset = target - position;
    final distance = offset.length;
    if (distance < 0.001) return;

    final direction = offset / distance;
    final step = math.min(speed * dt, distance);
    position += direction * step;
    angle = math.atan2(direction.y, direction.x);
  }
}

Worked example: anchor changes interpretation

A 40×20 component at (100, 100) with Anchor.topLeft occupies roughly x=100…140 and y=100…120 before rotation. With Anchor.center, the same position describes its center, so it occupies roughly x=80…120 and y=90…110. Center anchors usually simplify characters, projectiles, and rotations.

Common pitfall: mixing spaces

A pointer event may provide screen or local coordinates while an enemy lives in world coordinates viewed through a moving camera. Comparing them directly produces an offset that changes as the camera moves. Convert deliberately, or handle input on a component where Flame already supplies an appropriate local position.

Checkpoint

  1. What does subtracting one position from another produce?
  2. Why normalize before applying a constant speed?
  3. What happens to a child when its parent rotates?
Answer and discussion

Subtraction produces the offset from one point to the other. Normalization removes distance from the vector so all directions receive the same speed. A child inherits the parent transform, so its position and orientation are transformed with the parent.

Practice: transformation laboratory

Draw three rectangles with top-left, center, and bottom-right anchors. Place them at the same position, then rotate and scale them. Add a seeker that stops exactly at a clicked target without overshooting. Label world and local coordinates on screen.

Takeaways

  • Vector meaning comes from names and coordinate space.
  • Normalize direction, then apply speed and time.
  • Anchor is the reference for position, rotation, and scale.

C06 · Block 1 · Flame essentials

Assets, sprites, and animations

Declare resources explicitly, load them before they are needed, and turn image regions into time-based animation.

Learning objectives

  • Organize and declare image assets correctly.
  • Explain the relationship between image, sprite, sprite sheet, and animation.
  • Preload reusable graphics and avoid first-use stutter.

Mental model: bytes → image → region → frames

An asset file becomes a decoded image in memory. A sprite describes a region of that image and how to draw it. A sprite sheet packs many regions into one image. A sprite animation selects frames over time. Loading and decoding are lifecycle work; choosing the next already-loaded frame is update work.

Preload

Load assets for the next scene before gameplay begins. A short intentional loading state is better than a random hitch during the first attack.

Reuse

Many components can share one decoded image or animation data. Do not decode the same file for every enemy.

import 'package:flame/components.dart';
import 'package:flame/flame.dart';

class AnimatedPlayer extends SpriteAnimationComponent {
  AnimatedPlayer({required super.position})
      : super(
          size: Vector2.all(64),
          anchor: Anchor.center,
        );

  @override
  Future<void> onLoad() async {
    await super.onLoad();

    final image = await Flame.images.load('player_walk.png');
    animation = SpriteAnimation.fromFrameData(
      image,
      SpriteAnimationData.sequenced(
        amount: 6,
        stepTime: 0.10,
        textureSize: Vector2(32, 32),
      ),
    );
  }
}

Worked example: six-frame walk cycle

At stepTime: 0.10, each frame lasts one tenth of a second, so six frames complete in 0.6 seconds. The component may render at 64×64 while each source frame is 32×32; source texture size and world size are separate decisions.

Common pitfall: path and frame assumptions

Flame's image cache commonly resolves names under assets/images, while Flutter still requires the asset directory in pubspec.yaml. A single wrong filename, case difference, padding pixel, or frame count can make correct animation code look broken. Inspect the sheet dimensions and calculate the frame grid.

Checkpoint

  1. What is the difference between a decoded image and a sprite?
  2. Why preload an attack sound or animation?
  3. Does changing a component's world size change the source frame dimensions?
Answer and discussion

An image is pixel data; a sprite describes how to draw all or part of it. Preloading moves I/O and decoding away from the moment of first use. World size scales rendering but does not redefine the texture region.

Practice: two-state character

Add idle and walk sheets, preload both, and switch animation according to velocity. Confirm that stopping returns to idle without recreating the component. Document the asset license and record the memory dimensions of each source image.

Takeaways

  • Declare, preload, cache, and reuse.
  • Texture coordinates and world dimensions solve different problems.
  • Animation state should follow gameplay state rather than own it.

C07 · Block 2 · Game mechanics

Touch, mouse, keyboard, and controls

Translate device events into gameplay intentions so one rule set can serve several control schemes.

Learning objectives

  • Handle keyboard and pointer input with current Flame event APIs.
  • Separate physical buttons from actions such as move, fire, and pause.
  • Support simultaneous directions and predictable focus behavior.

Mental model: device event → intent → action

The Space key, a touch button, and a gamepad trigger can all mean fire. Components should consume that intent rather than know every physical device. This makes remapping, accessibility options, replay, and automated tests substantially easier.

LayerExamplesResponsibility
Physical inputWASD, pointer down, joystick axisRead platform events.
IntentMove vector, fire pressed, pause requestedNormalize and combine controls.

lib/game/input_game.dartDart
import 'package:flame/events.dart';
import 'package:flame/game.dart';
import 'package:flutter/services.dart';

class InputGame extends FlameGame with KeyboardEvents {
  final moveIntent = Vector2.zero();

  @override
  KeyEventResult onKeyEvent(
    KeyEvent event,
    Set<LogicalKeyboardKey> keysPressed,
  ) {
    moveIntent
      ..setZero()
      ..x = (keysPressed.contains(LogicalKeyboardKey.keyD) ? 1 : 0) -
          (keysPressed.contains(LogicalKeyboardKey.keyA) ? 1 : 0)
      ..y = (keysPressed.contains(LogicalKeyboardKey.keyS) ? 1 : 0) -
          (keysPressed.contains(LogicalKeyboardKey.keyW) ? 1 : 0);

    if (moveIntent.length2 > 0) {
      moveIntent.normalize();
    }
    return KeyEventResult.handled;
  }
}

Worked example: diagonal speed

Pressing D produces (1, 0), whose length is 1. Pressing D+S produces (1, 1), whose length is about 1.414. Normalizing prevents diagonal movement from being 41% faster. The player later multiplies the normalized intent by speed and dt.

Common pitfall: acting only on key-down

A key-down event is good for a one-shot action, but continuous movement should reflect the current pressed-key set. Otherwise release events, simultaneous keys, and lost focus can leave movement stuck. Also decide whether an input event should propagate to components underneath an overlay.

Checkpoint

  1. Why should the player not directly check whether Space is held?
  2. Why normalize a combined movement vector?
  3. Which actions are continuous and which are edge-triggered?
Answer and discussion

Consuming a fire intent decouples the player from hardware and enables remapping. Normalization preserves equal speed in all directions. Movement is commonly continuous; jump, confirm, pause, and a single shot are usually triggered on a press edge, though design can vary.

Practice: two control schemes

Create one movement-intent object. Feed it from WASD/arrow keys and from an on-screen joystick or drag gesture. The player must contain only one movement implementation. Add a pause overlay that captures input so gameplay does not continue receiving taps.

Takeaways

  • Map physical input to semantic intent.
  • Use current key state for continuous actions.
  • Normalize combined direction and manage focus explicitly.

C08 · Block 2 · Game mechanics

Movement, time, spawning, and randomness

Treat time as data, centralize cooldowns, and make random behavior reproducible when debugging.

Worked decision: built-in collisions or physics?

A top-down coin pickup needs overlap detection only, so Flame hitboxes are ideal. A crate that must fall, rotate, stack, and transfer momentum needs a physics solver, so use Forge2D. lib/game/adventure_game.dartDart
import 'package:flame/game.dart';

enum GamePhase { ready, playing, paused, won, gameOver }

class GameSession {
  int score = 0;
  int health = 3;

  void reset() {
    score = 0;
    health = 3;
  }
}

class AdventureGame extends FlameGame {
  final session = GameSession();
  GamePhase phase = GamePhase.ready;

  void startRound() {
    if (phase != GamePhase.ready) return;
    phase = GamePhase.playing;
  }

  void collectCoin() {
    if (phase != GamePhase.playing) return;
    session.score += 10;
    if (session.score >= 100) phase = GamePhase.won;
  }

  void damagePlayer() {
    if (phase != GamePhase.playing) return;
    session.health -= 1;
    if (session.health <= 0) phase = GamePhase.gameOver;
  }

  void pauseRound() {
    if (phase != GamePhase.playing) return;
    phase = GamePhase.paused;
    pauseEngine();
  }

  void resumeRound() {
    if (phase != GamePhase.paused) return;
    phase = GamePhase.playing;
    resumeEngine();
  }

  void prepareRestart() {
    session.reset();
    phase = GamePhase.ready;
  }
}

Worked example: what owns health?

Rule layer

Clamps health, applies invulnerability, decides game over, and emits a state change.

Presentation layer

Animates hearts, flashes the screen, and opens a game-over overlay. It does not decide the rule.

Common pitfall: boolean soup

isPaused, isDead, isWinning, and isStarting can accidentally all become true. A phase enum makes impossible combinations unrepresentable. Also remember that pauseEngine() stops game updates; the resume action must come from Flutter UI, an external lifecycle callback, or another mechanism that still runs while paused.

Checkpoint

  1. Why should a HUD not be the authoritative owner of score?
  2. What benefit does a transition table provide?
  3. Is an enemy's “stunned” state necessarily the same as the game's “paused” phase?
Answer and discussion

UI can be rebuilt or replaced, so durable game data belongs in a model. A transition table exposes legal and missing paths before code becomes tangled. “Stunned” is local to one enemy; “paused” is a session-wide phase. They operate at different levels and should be modelled separately.

Practice: design a complete round

Draw a transition table for ready, playing, paused, won, and gameOver. Implement command methods that reject illegal transitions. Then write three small tests: pausing from ready fails, health reaching zero ends play, and restart resets session data.

Takeaways

  • Use named phases for mutually exclusive global states.
  • Keep one authoritative session model and expose intention-revealing commands.
  • Separate rules from visual feedback and external services.

  }

Mental model: sense, decide, act

import 'package:flame/components.dart';

import 'adventure_game.dart';

enum EnemyState { patrol, chase, attack, stunned }

class Enemy extends PositionComponent
    with HasGameReference<AdventureGame> {
  Enemy({required super.position});

  EnemyState state = EnemyState.patrol;
  double speed = 90;
  double attackCooldown = 0;

  @override
  void update(double dt) {
    super.update(dt);
    attackCooldown -= dt;
    if (attackCooldown < 0) attackCooldown = 0;

    final offset = game.player.position - position;
    final distance = offset.length;

    // Use a larger exit radius to prevent rapid state flapping.
    if (state == EnemyState.patrol && distance < 180) {
      state = EnemyState.chase;
    } else if (state == EnemyState.chase && distance > 230) {
      state = EnemyState.patrol;
    } else if (state == EnemyState.chase && distance < 42) {
      state = EnemyState.attack;
    } else if (state == EnemyState.attack && distance > 58) {
      state = EnemyState.chase;
    }

    switch (state) {
      case EnemyState.patrol:
        patrol(dt);
      case EnemyState.chase:
        if (distance > 0) {
          position += offset.normalized() * speed * dt;
        }
      case EnemyState.attack:
        if (attackCooldown == 0) {
          game.tryEnemyAttack(this);
          attackCooldown = 1.2;
        }
      case EnemyState.stunned:
        break;
    }
  }

  void patrol(double dt) {
    // Follow the next authored waypoint.
  }
}

Worked difficulty tuning

Easy

Long telegraph, slower projectile, forgiving detection, one threat at a time.

Normal

Moderate cooldowns and overlapping enemy roles with readable gaps.

Hard

Shorter openings and smarter coordination—not invisible damage or ignored rules.

Keep these values in configuration records so difficulty changes data rather than duplicating AI code.

Common pitfall: state flapping

If chase starts below 180 pixels and stops above the same 180 pixels, tiny movement around the boundary can switch state every frame. Use hysteresis: enter at 180, exit at 230. State timers, explicit enter/exit methods, and a visible debug label make AI much easier to reason about.

Checkpoint

  1. Why split perception, decision, and action?
  2. What problem does hysteresis solve?
  3. What makes a difficult attack feel fair?
Answer and discussion

Separation makes each part testable and prevents movement code from secretly changing state. Hysteresis stops noisy transitions near a threshold. A fair attack is consistent, telegraphed, avoidable with learned skill, and governed by the same visible rules each time.

Takeaways

  • A small FSM makes mutually exclusive behavior explicit.
  • Transition thresholds, timers, and telegraphs shape fairness.
  • Difficulty is a data and encounter-design problem, not simply more health.

class SaveRepository {
  SaveRepository({SharedPreferencesAsync? preferences})
      : _preferences = preferences ?? SharedPreferencesAsync();

  static const _key = 'player_save_v1';
  final SharedPreferencesAsync _preferences;

  Future<PlayerSave> load() async {
    final raw = await _preferences.getString(_key);
    if (raw == null) return const PlayerSave();

    try {
      final decoded = jsonDecode(raw);
      if (decoded is! Map<String, dynamic>) {
        return const PlayerSave();
      }
      return PlayerSave.fromJson(decoded);
    } on FormatException {
      return const PlayerSave();
    }
  }

  Future<void> save(PlayerSave value) {
    return _preferences.setString(
      _key,
      jsonEncode(value.toJson()),
    );
  }
}

Worked save lifecycle

app starts | v load + validate + migrate | v construct session and level select | +-- settings change ----> debounce save +-- level completed ----> save checkpoint +-- app backgrounds ----> best-effort save The next launch always has safe defaults if data is absent or invalid.

Common pitfalls: every-frame writes and unversioned data

Disk-backed writes inside update() waste resources and can corrupt the player's experience when interrupted. Queue or debounce them.   3. Why use a repository rather than calling preferences from every component?

  • Persist compact domain data and reconstruct runtime components.

Advanced rendering

Rendering is the art of deciding what to draw, in what coordinate space, in what order, and at what cost. The best effect is not the most complicated one; it is the one that communicates gameplay clearly inside the frame budget.

Learning objectives

  • Reason about local transforms, component priority, compositing, and camera output.
  • Choose between direct drawing, cached snapshots, decorators, and post-processing.
  • Build a custom visual component without allocating new paint objects every frame.

Mental model: a pipeline, not a pile of pictures

Flame traverses the component tree, applies transforms, and renders siblings according to their priority. A component's render method normally draws in its own local coordinate system. The camera then presents a view of the world through its viewport. Effects that require the completed image—blur, color grading, bloom, distortion—belong later in the pipeline.

import 'dart:ui';
import 'package:flame/components.dart';

class PulseRing extends PositionComponent {
  PulseRing({required super.position})
      : super(size: Vector2.all(96), anchor: Anchor.center);

  final Paint _fill = Paint()..color = const Color(0x3329D9FF);
  final Paint _line = Paint()
    ..color = const Color(0xFF8BE9FD)
    ..style = PaintingStyle.stroke
    ..strokeWidth = 3;

  double phase = 0;

  @override
  void update(double dt) {
    phase = (phase + dt) % 1;
  }

  @override
  void render(Canvas canvas) {
    final center = (size / 2).toOffset();
    final radius = 24 + 18 * phase;
    _line.color = const Color(0xFF8BE9FD).withValues(alpha: 1 - phase);
    canvas.drawCircle(center, radius, _fill);
    canvas.drawCircle(center, radius, _line);
  }
}

Escalate effects deliberately

Begin with a sprite tint, scale pulse, or simple shape. Cache an unchanged complex subtree only after measurement shows repeated drawing is expensive. Use PostProcessComponent for a local subtree or CameraComponent.postProcess for a whole camera. Flame also distinguishes parallel and sequential post-process groups; in a sequential chain, every stage consumes the previous stage's output.

Official references: Flame post-processing, layers and snapshots, and component priority and render contexts.

Pitfall · Visual sophistication without visual meaning

Full-screen blur, particles, and screen shake can obscure hazards and create motion discomfort. Add an effects-intensity option, keep critical silhouettes readable, and never use an effect merely because the engine supports it.

Checkpoint

  1. Why does changing priority not move a component between coordinate spaces?
  2. When is a snapshot a poor choice?
  3. Where would you apply an effect that must color-grade the entire scene?
Open the discussion

Priority controls ordering among siblings, while parentage controls inherited transforms and context. A snapshot is poor for content that changes every frame because it must be regenerated.   • Effects serve feedback, clarity, and mood—not the other way around.

A useful scope sentence

“One polished five-minute run with one character, one arena, three meaningful hazards, and one complete restart flow” is testable. “A space roguelike with lots of enemies and progression” is an aspiration. Give every must-have a visible completion condition.

Pitfall · Designing the sequel first

Accounts, multiplayer, procedural campaigns, elaborate inventories, and multiple worlds multiply risk. Put good ideas in not now. Cutting scope protects the core; it does not betray it.

Checkpoint

  1. How does a pillar help reject a feature?
  2. What is the difference between a core loop and a feature list?
  3. Which risk should you prototype first?
Open the discussion

A pillar provides a decision rule: a feature that reinforces none of the pillars is likely noise. A loop describes repeated player decisions and feedback; a feature list merely names systems. Prototype the uncertainty that could invalidate the project or force the largest redesign.

Practice · Approve your own game

Write a one-page brief like the example. Limit yourself to three pillars, five must-haves, and a five-to-ten-minute target session. Add three risks with experiments that each take no more than one study session.

Takeaways

  • A clear promise and loop guide thousands of small decisions.
  • “Not now” is a production tool.
  • Prototype uncertainty before producing volume.

Chapter C24 · Final project

Prototype and vertical slice

A prototype asks whether the idea can work. A vertical slice asks whether you can deliver the final-quality experience in miniature. Confusing the two produces beautiful experiments or enormous unfinished games.

Learning objectives

  • Build throwaway prototypes around risky questions.
  • Define a thin end-to-end slice with production-quality standards.
  • Use exit criteria to decide when to continue, change direction, or stop.

Mental model: horizontal breadth versus vertical depth

A horizontal prototype touches many systems shallowly. A vertical slice follows one narrow player journey all the way down: input, rules, feedback, audio, UI, persistence, performance, and restart. The slice becomes the quality reference and reveals the real cost per unit of content.

PROTOTYPE: controls ─ combat ─ upgrades ─ map ─ enemies (broad, rough) VERTICAL SLICE: START ↓ PLAY ONE LOOP ↓ FEEDBACK + AUDIO + HUD ↓ RESULT → SAVE → RESTART (narrow, deep)
lib/game/build_scope.dartDart · explicit slice boundary
enum BuildScope { prototype, verticalSlice, release }

class GameConfig {
  const GameConfig({required this.scope});
  final BuildScope scope;

  bool get enableMetaProgression => scope == BuildScope.release;
  bool get useFinalFeedback => scope != BuildScope.prototype;
}

const config = GameConfig(scope: BuildScope.verticalSlice);
Slice gatePass condition
Playable loopA new player can start, act, win or lose, and restart without developer help.
Representative qualityOne environment and one interaction meet the intended art, sound, and feedback bar.
Technical healthNo known save corruption or blocker; target scene meets its measured performance budget.
LearningPlaytest evidence answers the largest design risk.

Slice backlog

Write work as observable outcomes: “Player can restart from the result screen with keyboard and touch” is better than “finish UI.” Mark everything outside the chosen journey as deferred, even if its code would be interesting.

Pitfall · Polishing before the loop survives

Do not spend a day on particles while the core decision is still dull. Prototype with disposable assets. Raise the quality bar only after the risky interaction has earned another iteration.

Checkpoint

  1. What question does a prototype answer?
  2. Why must a vertical slice include restart and failure?
  3. When should prototype code be discarded?
Open the discussion

A prototype answers a named uncertainty, not “is the whole game good?” Failure and restart complete the real player loop and expose navigation/state problems. Discard code when its shortcuts make the production architecture harder to trust; preserve the learning, not every line.

Practice · One complete minute

Define and build a 60–90 second journey from launch to restart. Write four pass conditions before coding. Play it three times with debug controls disabled, then cut anything that does not strengthen the central decision.

Takeaways

  • Prototype risks cheaply; slice one experience deeply.
  • Exit criteria prevent endless “almost ready.”
  • The slice estimates quality, architecture, and content cost together.

Chapter C25 · Final project

Content, balance, polish, and QA

Once the slice works, production becomes controlled variation: create content from proven patterns, tune with evidence, improve feedback, and systematically search for ways the build can fail.

Learning objectives

  • Move tuning values out of scattered component code.
  • Balance with hypotheses, telemetry notes, and playtest observation.
  • Run a risk-based QA matrix and manage bugs by severity.

Mental model: change one lever, observe one outcome

Balance is not the pursuit of mathematically equal options. It is the shaping of interesting decisions for the intended audience. Establish a baseline, name the problem, change one lever, observe multiple players, and keep or revert. Polish improves how clearly the game communicates state; QA challenges the assumptions behind that state.

  "score":  { "pickup": 100, "comboWindowSeconds": 2.0 }

QA axisRepresentative cases
StateFirst launch, continue, pause, resume, win, lose, restart, corrupted/missing save.
TimingRapid taps, simultaneous collisions, long idle, low/high frame rate, background/resume.
Display/inputSmall and large window, resize, touch, keyboard, mouse, focus loss.
ContentEvery enemy, level edge, upgrade combination, maximum values, empty values.

Write actionable bug reports

Include build and platform, starting state, exact steps, expected result, actual result, reproduction rate, and evidence. Classify severity by player impact and priority by when the team should act. A rare save-destroying defect can be more severe than a frequent visual glitch.

Pitfall · Tuning from your own mastery

You know every rule and have practiced the controls for hours. New players do not. Watch silently: where do they look, hesitate, misread, or quit? Do not teach around a problem that the game itself should explain.

Checkpoint

  1. Why change one balance lever at a time?
  2. How is polish different from feature growth?
  3. What is the difference between severity and priority?
Open the discussion

One-variable experiments make cause and effect easier to infer. Polish clarifies and strengthens existing interactions; feature growth creates new systems and risk. Severity describes impact, while priority combines impact with schedule, reach, cost, and release needs.

Practice · Three-player tuning pass

Choose one measurable concern, such as “the first minute overwhelms beginners.” Record the current value, predicted outcome, and three observations. Change one value, repeat the same scenario, then write a decision. Finish with a ten-case regression checklist.

Takeaways

  • Centralized tuning data accelerates safe iteration.
  • Observation beats designer intuition alone.
  • QA is structured risk discovery plus disciplined regression.

Chapter C26 · Final project

Building, signing, and publishing

A release is a reproducible artifact plus the evidence, metadata, permissions, and ownership needed to put it safely in a player's hands.

Learning objectives

  • Prepare versioned release builds for selected Flutter targets.
  • Keep signing credentials out of source control.
  • Run a release checklist covering behavior, legal assets, store material, and rollback.

Mental model: source → signed artifact → channel → player

A build compiles one reviewed source state into a platform artifact. Signing binds that artifact to a publisher identity where the platform requires it. A store or distribution channel adds review, metadata, policy, and staged delivery. Each released artifact must be traceable back to a version and commit.

Release gateEvidence
IdentityUnique application ID, semantic version, increasing build number, matching release notes.
QualityClean analysis/tests, release-mode smoke test, supported-device matrix, no blocker defects.

PresentationIcon, screenshots, description, age/content rating, support and privacy links.

Target-specific work remains

Flutter supplies platform build commands, but stores have distinct certificates, signing files, packaging rules, screenshots, and review processes. Android commonly distributes an app bundle; iOS signing and App Store submission require Apple's toolchain; Windows and web have their own packaging and hosting choices. Follow the current guide for the chosen target immediately before release.

Official references: Flutter's deployment hub and Android release guide.

Pitfall · Treating secrets as project assets

Never commit keystores, passwords, private keys, API secrets, or populated signing-property files. Ignore them, store them in a secret manager or protected local location, document how authorized maintainers obtain them, and keep a secure backup. Losing a signing identity can block future updates.

Checkpoint

  1. What do the two parts of 1.0.0+1 represent?
  2. Why smoke-test the produced release artifact instead of only debug mode?
  3. Why must every release map to a commit?
Open the discussion

The first part is the user-facing semantic version; the suffix is the platform build number and should increase for submissions. Release builds can differ in compilation, assertions, assets, permissions, and signing. Commit traceability lets you reproduce, diagnose, patch, and audit exactly what shipped.

Practice · Release candidate zero

Choose one target. Create a release checklist, bump to 0.1.0+1, run formatting/analyzer/tests, build the artifact, and launch that artifact outside the IDE. Archive the checklist and record its commit hash, size, platform, date, and three smoke-test results. Do not publish publicly yet.

Takeaways

  • Build, signing, and distribution are separate responsibilities.
  • Protect publisher credentials and make builds traceable.
  • The real release candidate is the artifact players receive.

Chapter C27 · Final project

Portfolio and retrospective

Shipping proves persistence. A strong case study proves judgment: what you tried, how the system works, which evidence changed your mind, and what you would do differently.

Learning objectives

  • Present a playable project with a concise technical narrative.
  • Document architecture, constraints, testing, and measured decisions.
  • Run a blameless retrospective that produces concrete next actions.

Mental model: claim, evidence, reflection

“I built a game” is a claim. A playable link or artifact, focused video, source excerpt, test results, and performance capture are evidence. A retrospective explains your reasoning and growth. Together they reveal more than a long feature inventory.

Retrospective prompts

Look back

What was the original promise? Which evidence shows it was or was not achieved? Where did time actually go?

Look forward

What should be repeated, stopped, or changed? Which single skill would most improve the next project?

Write about decisions you personally made. Credit collaborators and asset creators precisely. Remove credentials, private analytics, and third-party assets you cannot redistribute before sharing a repository.

Pitfall · A portfolio that makes the reviewer investigate

Do not hide the build below a wall of prose, require undocumented setup, or show ten minutes of menus before gameplay. Lead with the result, make controls obvious, state your contribution, and keep deep technical material available for readers who want it.

Checkpoint

  1. What evidence supports a performance claim?
  2. Why state your personal contribution explicitly?
  3. What makes a retrospective actionable?
Open the discussion

A reproducible scenario, profile capture, numbers, and before/after comparison support a performance claim. Clear ownership prevents misleading credit and helps reviewers ask relevant questions. An actionable retrospective turns observations into specific experiments or habits with an owner and next opportunity.

Practice · Publish the story

Create the README, one architecture diagram, a short gameplay capture, and a release link or exact artifact instructions. Ask someone unfamiliar with the project to find the build and controls in under 30 seconds. Finish with three retrospective actions: repeat, stop, and try next.

Takeaways

  • Curate evidence around the experience and your decisions.
  • Make the project effortless to understand and try.
  • A finished game's lessons are raw material for the next finished game.
